# VK Section Rules

Правила раздела ВК на текущем этапе подготовки базы.

## Статус
- Документ живой и обновляемый.
- Текущая версия не считается окончательной спецификацией.

## Подтверждённые требования
1. Система в будущем должна принимать Word-файлы с реестрами.
2. Данные из реестров должны извлекаться и нормализоваться.
3. Данные должны раскладываться по Excel-таблицам.
4. Даты должны подставляться по журналу работ.
5. Материалы и используемые листы должны учитываться.
6. Сертификаты должны привязываться к материалам.
7. Если на акт 6 и более сертификатов, нужно формировать приложение.
8. Акты должны формироваться по Word-шаблонам.
9. Выход: один акт = один Word-файл.
10. В дальнейшем подход должен расширяться на OV1 и OV2.
11. Состав актов брать строго из реестра; не отклоняться, кроме явной грамматической опечатки.
12. При конфликте приоритет у примечаний в реестре.
13. Журнал используется для сверки; отклонения дат 1-2 дня допустимы, если нет явных нестыковок.
14. Дата окончания работ должна быть не позже даты закрытия.
15. Именование акта: `№<номер>-<система>_<секция>.docx` с обязательным символом `№`.
16. Для приложений/испытаний/промывки и других спецтипов добавляется приписка в конце имени.
17. Сертификаты для формирования брать по реестру; дополнительные сертификаты в папке не обязательны к обработке.
18. Если сертификатов 6 и более — делать приложение; если 5 и меньше — реквизиты писать в акте.
19. Допустима единая рабочая таблица с разбиением по секциям сверху вниз по реестрам.
20. Схемы пользователь готовит вручную; текущий контур автоматизации на них не распространяется.
21. Формат дат зависит от конкретного места в документе и берётся по примерам.
22. Исходные акты в `inputs/` полностью read-only: запрещено менять содержимое и имена файлов.
23. Любые проверки, переименование и нормализация выполнять только на копиях в `output/`.
24. В `output/vk/projects/<project>/sections/<section>/` готовые копии актов раскладывать по системам (типам актов) в отдельные папки.
25. Для групп с вариативным типом в отдельных строках (например, акты `№25-№28` внутри группы `В1.1,Т3.1,Т4.1/1`) использовать общую папку группы по заголовку реестра.
26. В `output` не копировать `shared`-контент из `inputs`; в выходе оставлять только структуру с актами по секциям.
27. Лишние акты (без пары в реестре) не раскладывать по системным папкам: помещать в корень секции и ставить префикс `! ` в начале имени.
28. После `! ` указывать номер акта из содержимого файла; если тип акта распознан, добавлять его тег в имени.
29. При дубликатах одного номера основной номер остаётся у акта, совпадающего с реестром по заголовку; дополнительный акт включается с суффиксом номера (`№6.1-...`) в той же системной папке.
30. Для дубликатных актов добавлять отдельную строку в `Реестр (акты)` сразу под базовым номером и выделять такие строки ярко-синим.
31. Эталон названия объекта, ФИО подписантов, номеров приказов и реквизитов организаций для шаблонов — существующие акты `Промывка` и `Дезинфекция`.
32. АОУСИТО не используется для проверки этих данных.
33. В листе `База актов` колонка `Последующая работа` заполняется только для строк `АОСР`.
34. Для `АОСР` в колонку `Последующая работа` подставляется наименование следующего акта по порядку строк на листе `База актов`.
35. Если следующий акт не `АОСР`, в колонку `Последующая работа` записывается название следующей работы в нормализованной форме действия, а не буквальный заголовок `Акт ...` (например, `Гидростатическое испытание ...`, `Проведение промывки ...`, `Освидетельствование ...`).

## Границы раздела
- Особенности ВК описываются в `sections/VK/`.
- Нельзя переносить ВК-специфику в общее ядро без подтверждения универсальности.

## Операционные правила
- При нехватке данных не придумывать логику заполнения полей.
- Все спорные случаи фиксировать в `docs/open-questions.md`.
- При изменении логики сразу обновлять документы в `docs/` и `sections/VK/`.
