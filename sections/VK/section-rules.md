# VK Section Rules

Правила раздела ВК на текущем этапе подготовки базы.

## Статус
- Документ живой и обновляемый.
- Текущая версия не считается окончательной спецификацией.

## Подтверждённые требования
1. Система в будущем должна принимать Word-файлы с реестрами.
2. Данные из реестров должны извлекаться и нормализоваться.
3. Данные должны раскладываться по Excel-таблицам.
4. Даты должны подставляться по журналу работ.
5. Материалы и используемые листы должны учитываться.
6. Сертификаты должны привязываться к материалам.
7. Если на акт 6 и более сертификатов, нужно формировать приложение.
8. Акты должны формироваться по Word-шаблонам.
9. Выход: один акт = один Word-файл.
10. В дальнейшем подход должен расширяться на OV1 и OV2.
11. Состав актов брать строго из реестра; не отклоняться, кроме явной грамматической опечатки.
12. При конфликте приоритет у примечаний в реестре.
13. Журнал используется для сверки; отклонения дат 1-2 дня допустимы, если нет явных нестыковок.
14. Дата окончания работ должна быть не позже даты закрытия.
15. Именование акта: `№<номер>-<система>_<секция>.docx` с обязательным символом `№`.
16. Для приложений/испытаний/промывки и других спецтипов добавляется приписка в конце имени.
17. Сертификаты для формирования брать по реестру; дополнительные сертификаты в папке не обязательны к обработке.
18. Если сертификатов 6 и более — делать приложение; если 5 и меньше — реквизиты писать в акте.
19. Допустима единая рабочая таблица с разбиением по секциям сверху вниз по реестрам.
20. Схемы пользователь готовит вручную; текущий контур автоматизации на них не распространяется.
21. Формат дат зависит от конкретного места в документе и берётся по примерам.
22. Исходные акты в `inputs/` полностью read-only: запрещено менять содержимое и имена файлов.
23. Любые проверки, переименование и нормализация выполнять только на копиях в `output/`.
24. В `output/vk/projects/<project>/sections/<section>/` готовые копии актов раскладывать по системам (типам актов) в отдельные папки.
25. Для групп с вариативным типом в отдельных строках (например, акты `№25-№28` внутри группы `В1.1,Т3.1,Т4.1/1`) использовать общую папку группы по заголовку реестра.
26. В `output` не копировать `shared`-контент из `inputs`; в выходе оставлять только структуру с актами по секциям.
27. Лишние акты (без пары в реестре) не раскладывать по системным папкам: помещать в корень секции и ставить префикс `! ` в начале имени.
28. После `! ` указывать номер акта из содержимого файла; если тип акта распознан, добавлять его тег в имени.
29. При дубликатах одного номера основной номер остаётся у акта, совпадающего с реестром по заголовку; дополнительный акт включается с суффиксом номера (`№6.1-...`) в той же системной папке.
30. Для дубликатных актов добавлять отдельную строку в `Реестр (акты)` сразу под базовым номером и выделять такие строки ярко-синим.
31. Эталон названия объекта, ФИО подписантов, номеров приказов и реквизитов организаций для шаблонов — существующие акты `Промывка` и `Дезинфекция`.
32. АОУСИТО не используется для проверки этих данных.
33. В листе `База актов` колонка `Последующая работа` заполняется только для строк `АОСР`.
34. Для `АОСР` в колонку `Последующая работа` подставляется наименование следующего акта по порядку строк на листе `База актов`.
35. Если следующий акт не `АОСР`, в колонку `Последующая работа` записывается название следующей работы в нормализованной форме действия, а не буквальный заголовок `Акт ...` (например, `Гидростатическое испытание ...`, `Проведение промывки ...`, `Освидетельствование ...`).
36. Для `АОСР` дата начала работ по системе не должна быть раньше даты начала работ соответствующего `АОУСИТО`.
37. Перед использованием даты начала работ `АОУСИТО` из примечания её нужно сверить с существующим актом `АОУСИТО`; дата в примечании должна совпадать с датой в акте.
38. Если после применения нижней границы `АОУСИТО` не удаётся найти прямую привязку к журналу, даты `АОСР` не придумываются: строка остаётся на ручную проверку, а пояснение добавляется в `Примечания`.
39. Для серий `АОСР` по однотипным работам нельзя считать порядок строк или этажей порядком фактического выполнения: допустимы параллельные группы актов с одинаковыми диапазонами дат.
40. При заполнении дат по журналу для таких серий нужно учитывать групповые и параллельные выполнения работ; отсутствие строгой монотонности по этажам само по себе не считается ошибкой.
41. Рабочее распределение дат перед переносом в итоговую базу ведётся на отдельном листе `Даты работ`.
42. На текущем этапе бот по журналу ищет и предлагает только дату начала работ; даты окончания пользователь проставляет вручную.
43. Зелёные даты на листе `Даты работ`, проставленные пользователем вручную, не редактируются ботом.
44. Если пользовательские даты противоречат пределам соответствующего `АОУСИТО` или дата окончания позже даты оформления акта, значения не переписываются, а только выделяются оранжевым.
45. Диапазон в колонке `Пределы по АОУСИТО` берётся из существующего акта `АОУСИТО` по соответствующему разделу/системе, а не из эвристики.
46. При поиске даты начала работ по журналу записи о бурении отверстий, сверлении и аналогичной подготовке не считаются подтверждением монтажа трубопроводов или других элементов системы.
47. Если для текущего проекта в `output/.../work-log/` есть вручную сверенный производный журнал, при заполнении `Даты работ` он используется как основной текстовый источник для строк журнала; сырая копия из `inputs/` остаётся только исходником для повторной проверки.
48. Для серий `АОСР` по монтажу горизонтальной разводки из сшитого полиэтилена базовое допущение: этажи идут почти линейно, без активного распараллеливания; параллельность для таких серий считать редким исключением, которое нужно подтверждать отдельными признаками.
49. Если между двумя опорными записями журнала по такой серии можно разумно распределить этажи, диапазоны дат раскладываются по этажам более-менее равномерно.
50. Если при равномерной раскладке получается более 5 дней на этаж, весь разрыв не растягивается: вместо этого применяется локальный проектный тренд предыдущих подтверждённых участков серии, обычно около 2-3 дней на этаж.
51. Если опорная запись журнала относится к смешанному разделу `ОВ ВК` и не содержит явного указания систем ВК, такую запись нельзя считать прямой сверкой для `ВК`: она может использоваться только как косвенный якорь для рабочей гипотезы на листе `Даты работ`.
52. Эту логику линейной раскладки по горизонталям из сшитого полиэтилена для текущего ВК считать подтверждённой пользовательской практикой; в будущем её можно переиспользовать для раздела `ОВ1`, но без преждевременного переноса ВК-специфики в общее ядро.
53. Пользовательское наблюдение по следующим этапам для будущей работы: антикоррозия и изоляция идут после монтажа; типовой резерв на каждый такой этап — 2-3 дня, а для маленькой системы возможен 1 день.
54. После ручного завершения лист `Даты работ` считается зафиксированным чекпоинтом и больше не редактируется; дальнейший перенос дат в `База актов` выполняется только при точном совпадении строки по полям `Секция`, `№ акта`, `Тип акта` и `Наименование акта`.
55. Для подготовки материалов по `АОСР` используется отдельный рабочий лист `Материалы` в облегчённом формате для парной проверки (`CONFIRMED`, 2026-03-28).
56. В листе `Материалы` не дублируются раздельные реквизиты сертификатов (номер и сроки действия); детальная сверка выполняется по листу `Реестр (сертификаты)` (`CONFIRMED`, 2026-03-28).
57. Если для `АОСР` сертификатов более 5, в акте указываются материалы без реквизитов сертификатов, а сами реквизиты выносятся в отдельное приложение (`CONFIRMED`, 2026-03-28).
58. Имя файла приложения к `АОСР`: имя файла акта + суффикс ` прил` перед расширением `.docx` (`CONFIRMED`, 2026-03-28).
59. Входной комплект проекта допускается загружать целиком в `inputs/vk/projects/<project>/shared/source-project/`; последующее разделение PDF на графическую часть и спецификации выполняется как производная обработка (`CONFIRMED`, 2026-03-28).
60. На листе `Реестр (сертификаты)` после колонки `Секция` используется колонка `Название материала/оборудования (для акта)` для рабочего названия в акте (`CONFIRMED`, 2026-03-28).
61. При формировании текста для акта связка `материал + реквизиты сертификата` строится через сопоставление названия материала/оборудования с номером и сроками действия сертификата из `Реестр (сертификаты)` (`CONFIRMED`, 2026-03-28).
62. Один сертификат может относиться к нескольким материалам/видам оборудования; это отражается несколькими строками с одинаковыми реквизитами сертификата и разными значениями в колонке `Название материала/оборудования (для акта)` (`CONFIRMED`, 2026-03-28).
63. На листе `Материалы` колонка `Сертификаты (по реестру)` не используется; для рабочей проверки остаётся колонка `Материалы (для акта)` и сопоставление с `Реестр (сертификаты)` (`CONFIRMED`, 2026-03-28).
64. В формулировке материала двоеточие перед перечнем типоразмеров/моделей ставится только при множественном перечислении; для одного типоразмера двоеточие не ставится (`CONFIRMED`, 2026-03-28).
65. Для строк `трубопровод из труб стальных ...` и `гильзы из труб стальных ...` тип трубы определяется по диаметру: до `∅40` включительно — `ВГП`, от `∅57` включительно — `электросварные` (`CONFIRMED`, 2026-03-28).
66. В колонке `Приложение (если >5 сертификатов)` листа `Материалы` используются значения `Да/Нет`; значение `Да` подсвечивается синим (`CONFIRMED`, 2026-03-28).

## Границы раздела
- Особенности ВК описываются в `sections/VK/`.
- Нельзя переносить ВК-специфику в общее ядро без подтверждения универсальности.

## Операционные правила
- При нехватке данных не придумывать логику заполнения полей.
- Все спорные случаи фиксировать в `docs/open-questions.md`.
- При изменении логики сразу обновлять документы в `docs/` и `sections/VK/`.
