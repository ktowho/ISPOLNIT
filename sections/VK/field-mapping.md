# VK Field Mapping (Draft)

Черновая карта полей между входами и внутренней структурой для ВК.

## Важно
- Это не финальная таблица колонок.
- До получения реальных входов текущего проекта названия полей считаются рабочими гипотезами.
- При конфликте данных приоритет у примечаний реестра (`CONFIRMED`).
- Для полей шаблонов `object_name`, `signers`, `orders`, `org_requisites` эталон текущего проекта — существующие акты `Промывка` и `Дезинфекция` (`CONFIRMED`).

## 1) Реестр актов (Word) -> Act
1. `Номер акта` -> `act.act_number` (`HYPOTHESIS`)
2. `Система` -> `act.system_code` (`HYPOTHESIS`)
3. `Тип акта` -> `act.act_type` (`HYPOTHESIS`)
4. `Дата завершения работ` -> `act.date_work_end` (`HYPOTHESIS`)
5. `Дата подписания` -> `act.date_sign` (`HYPOTHESIS`)
6. `Листы ИС/схемы` -> `act.schemes[]` (`HYPOTHESIS`)
7. `Материалы` -> `act.materials[]` (`HYPOTHESIS`)
8. `Сертификаты` -> `act.certificates[]` (`HYPOTHESIS`)

## 2) Журнал работ -> WorkLogEntry
1. `Вид работ` -> `work_log.work_type` (`HYPOTHESIS`)
2. `Дата начала` -> `work_log.date_start` (`HYPOTHESIS`)
3. `Дата окончания` -> `work_log.date_end` (`HYPOTHESIS`)
4. `Привязка к системе/участку` -> `work_log.scope_ref` (`HYPOTHESIS`)

## 3) Реестр сертификатов/файлы сертификатов -> Certificate
1. `Наименование материала/оборудования` -> `certificate.material_links[]` (`HYPOTHESIS`)
2. `Номер/идентификатор сертификата` -> `certificate.certificate_id` (`HYPOTHESIS`)
3. `Файл сертификата` -> `certificate.certificate_file` (`CONFIRMED` как сущность файла)

## 4) Бизнес-правила связки (`CONFIRMED`/`OBSERVED`)
1. Даты актов сверяются с журналом работ.
2. Допустимы отклонения дат 1-2 дня, если нет явных нестыковок.
3. Дата окончания работ не должна быть позже закрытия.
4. Для `АОСР` дата начала работ по системе не может быть раньше даты начала работ соответствующего `АОУСИТО`.
5. Дата начала работ `АОУСИТО`, указанная в примечании рабочей таблицы, предварительно сверяется с существующим актом `АОУСИТО` и должна совпадать с датой в акте.
6. Для серий однотипных `АОСР` (например, горизонтальная разводка по этажам) допустимы параллельные группы работ и одинаковые диапазоны дат у нескольких соседних актов; порядок строк не равен порядку выполнения.
7. Материалы должны иметь связанный сертификат по реестру.
8. Если сертификатов на акт 6 и более, формируется приложение.
9. Если сертификатов 5 и меньше, реквизиты указываются в акте.
10. Выход формируется как отдельный Word-файл на каждый акт.
11. Именование файла: `№<номер>-<система>_<секция>.docx` плюс приписка типа при необходимости.
12. Исходные акты в `ready-acts` полностью read-only: не править содержимое и не переименовывать файлы в `inputs/`.

## 6) Рабочая таблица реестров (зафиксировано, 2026-03-28)
1. Рабочая таблица ведётся отдельно от журнала работ (`CONFIRMED`).
2. Рабочая таблица хранится как производный артефакт вне `inputs/`, в `output/spreadsheet/` (`CONFIRMED`).
3. Обязательные листы рабочей таблицы (`CONFIRMED`):
- `База актов`
- `Даты работ`
- `Реестр (акты)`
- `Реестр (схемы)`
- `Реестр (сертификаты)`
- `Материалы`
4. Лист `База актов` является главным итоговым листом для дальнейших срезов по типам актов и секциям (`CONFIRMED`).
5. Лист `Даты работ` используется как промежуточный рабочий лист для распределения дат по журналу до переноса подтверждённых значений в `База актов` (`CONFIRMED`, 2026-03-28).
6. Порядок колонок листа `Даты работ` слева направо (`CONFIRMED`, 2026-03-28):
- `Секция`
- `№ акта`
- `Тип акта`
- `Наименование акта`
- `Дата начала работ`
- `Дата завершения работ`
- `Дата оформления`
- `Пределы по АОУСИТО`
- `Комментарии бота`
- `Запись из журнала`
7. Порядок колонок листа `База актов` слева направо (`CONFIRMED`):
- `Шаблон`
- `Секция`
- `№ акта`
- `Тип акта`
- `Наименование акта`
- `Последующая работа`
- `Дата начала работ`
- `Дата завершения работ`
- `Дата оформления`
- `Примечания`
8. Правило заполнения колонки `Последующая работа` (`CONFIRMED`, 2026-03-28):
- колонка заполняется только для строк, где `Шаблон = АОСР`;
- значение берётся из наименования следующего акта по порядку строк на листе `База актов`;
- если следующий акт не `АОСР`, в колонку заносится не буквальный заголовок акта, а название следующей работы в форме действия;
- типовые нормализации: `Акт гидростатического испытания ...` -> `Гидростатическое испытание ...`, `Акт проведения промывки ...` -> `Проведение промывки ...`, `Акт о проведении испытания ...` -> `Проведение испытания ...`, `Акт освидетельствования ...` -> `Освидетельствование ...`.
9. Для листа `Даты работ` голубой заливкой отмечаются только те даты, которые бот предложил или дополнил самостоятельно, а не просто перенёс из существующей базы (`CONFIRMED`, 2026-03-28).
10. Для каждой строки на листе `Даты работ` в колонке `Пределы по АОУСИТО` выводится диапазон дат соответствующего акта `АОУСИТО` в формате `ДД.ММ.ГГГГ - ДД.ММ.ГГГГ` (`CONFIRMED`, 2026-03-28).
11. На текущем этапе бот по журналу ищет только даты начала работ; даты окончания на листе `Даты работ` пользователь ведёт вручную (`CONFIRMED`, 2026-03-28).
12. Зелёные даты, проставленные пользователем вручную на листе `Даты работ`, бот не редактирует; при противоречиях с `АОУСИТО` или датой оформления такие ячейки только выделяются оранжевым (`CONFIRMED`, 2026-03-28).
13. Значение колонки `Пределы по АОУСИТО` берётся из существующего акта `АОУСИТО` по соответствующей системе/разделу, а не вычисляется эвристически (`CONFIRMED`, 2026-03-28).
14. При поиске даты начала работ по журналу записи о бурении отверстий, сверлении и иной подготовке проёмов не считаются подтверждением монтажа системы (`CONFIRMED`, 2026-03-28).
15. Пользовательское наблюдение для будущих эвристик: антикоррозия и изоляция обычно идут после монтажа; типовой резерв на каждый этап составляет 2-3 дня, а для маленькой системы возможен 1 день (`OBSERVED`, 2026-03-28).
16. Порядок колонок листа `Реестр (акты)` слева направо (`CONFIRMED`):
- `Шаблон`
- `Секция`
- `№ акта`
- `Тип акта`
- `Наименование акта`
- `Дата оформления`

17. Допустимые значения поля `Шаблон` для текущего ВК (`CONFIRMED`):
- `АОСР`
- `АОУСИТО`
- `гидравлика`
- `дезинфекция`
- `испытания оборудования`
- `испытание на водоотдачу`
- `испытание АПТ`
- `пролив`
- `промывка`
18. Цветовая маркировка листа `Реестр (акты)` (`CONFIRMED`):
- `АОСР`: без заливки
- `АОУСИТО`: без заливки
- `дезинфекция`: светло-оранжевый
- `промывка`: светло-голубой
- `испытания оборудования`: светло-розовый
- `гидравлика`: светло-фиолетовый
- `пролив`: светло-жёлтый
- Готовые акты из `inputs/.../ready-acts/`: светло-зелёный с приоритетом над категорийной заливкой.
- Добавленные дубликаты номеров (например, `№6.1-...`): ярко-синий.
19. Для `АОСР` по горизонтальной разводке из сшитого полиэтилена (`OBSERVED` -> `CONFIRMED` пользователем, 2026-03-28):
- по умолчанию предполагается почти линейное продвижение по этажам, а не активная параллельная раскатка;
- между двумя опорными записями журнала этажи распределяются более-менее равномерно;
- если такой расчёт даёт более 5 дней на этаж, применяется локальный тренд проекта по предыдущим подтверждённым участкам серии, обычно 2-3 дня на этаж;
- смешанные строки журнала `ОВ ВК` без явного упоминания систем ВК использовать только как косвенную опору для рабочей гипотезы, а не как прямое подтверждение дат по ВК;
- эту эвристику пока держать в правилах ВК, но учитывать как кандидат на перенос в `ОВ1` после отдельного подтверждения.
20. После ручного подтверждения лист `Даты работ` становится фиксированным промежуточным слоем; перенос колонок `Дата начала работ` и `Дата окончания работ` в `База актов` выполняется только по точному совпадению `Секция + № акта + Тип акта + Наименование акта` (`CONFIRMED`, 2026-03-28).
21. Лист `Материалы` используется как рабочий лист парной проверки по `АОСР`: в нём фиксируются материалы для акта и комментарии по проверке без избыточной детализации сертификатов (`CONFIRMED`, 2026-03-28).
22. Порядок колонок листа `Материалы` слева направо (`CONFIRMED`, 2026-03-28):
- `Секция`
- `№ акта`
- `Тип акта`
- `Наименование акта`
- `Материалы (как в проекте)`
- `Материалы (для акта)`
- `Сертификаты (по реестру)`
- `Приложение (если >5 сертификатов)`
- `Статус проверки`
- `Комментарий`
23. Детальные реквизиты сертификатов (номер, дата начала и дата окончания действия) в листе `Материалы` не дублируются; источник этих данных — лист `Реестр (сертификаты)` (`CONFIRMED`, 2026-03-28).
24. Для `АОСР` при количестве сертификатов более 5 в основном тексте акта перечисляются материалы без реквизитов сертификатов; реквизиты выносятся в отдельное приложение (`CONFIRMED`, 2026-03-28).
25. Имя файла приложения к `АОСР` совпадает с именем файла акта, но с суффиксом ` прил` в конце названия перед `.docx` (`CONFIRMED`, 2026-03-28).

## 7) Выходные копии ready-acts (зафиксировано, 2026-03-27)
1. Все проверки и переименование готовых актов выполняются только на копиях в `output/vk/projects/<project>/...` (`CONFIRMED`).
2. В `output` формируется только рабочая структура секций с актами; `shared`-контент из `inputs` не дублируется (`CONFIRMED`).
3. Копии актов раскладываются по папкам систем (типов актов) согласно реестру; для вариативных строк внутри одной группы используется папка группы по заголовку реестра (`CONFIRMED`).
4. Если акту не найдена пара в реестре (лишний акт), он размещается в корне секции и в имени получает префикс `! `, затем номер из содержимого (с fallback на имя файла) и, при распознавании, тег типа акта (`CONFIRMED`).
5. Если по одному номеру найдено несколько исходных файлов, основной номер остаётся у акта, который совпадает с реестром по заголовку; дополнительный акт включается в комплект как дубликат с суффиксом номера (`№6.1-...`) и отдельной строкой в `Реестр (акты)` (`CONFIRMED`).
