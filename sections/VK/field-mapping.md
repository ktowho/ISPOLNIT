# VK Field Mapping (Draft)

Черновая карта полей между входами и внутренней структурой для ВК.

## Важно
- Это не финальная таблица колонок.
- До получения реальных входов текущего проекта названия полей считаются рабочими гипотезами.
- При конфликте данных приоритет у примечаний реестра (`CONFIRMED`).
- Для полей шаблонов `object_name`, `signers`, `orders`, `org_requisites` эталон текущего проекта — существующие акты `Промывка` и `Дезинфекция` (`CONFIRMED`).

## 1) Реестр актов (Word) -> Act
1. `Номер акта` -> `act.act_number` (`HYPOTHESIS`)
2. `Система` -> `act.system_code` (`HYPOTHESIS`)
3. `Тип акта` -> `act.act_type` (`HYPOTHESIS`)
4. `Дата завершения работ` -> `act.date_work_end` (`HYPOTHESIS`)
5. `Дата подписания` -> `act.date_sign` (`HYPOTHESIS`)
6. `Листы ИС/схемы` -> `act.schemes[]` (`HYPOTHESIS`)
7. `Материалы` -> `act.materials[]` (`HYPOTHESIS`)
8. `Сертификаты` -> `act.certificates[]` (`HYPOTHESIS`)

## 2) Журнал работ -> WorkLogEntry
1. `Вид работ` -> `work_log.work_type` (`HYPOTHESIS`)
2. `Дата начала` -> `work_log.date_start` (`HYPOTHESIS`)
3. `Дата окончания` -> `work_log.date_end` (`HYPOTHESIS`)
4. `Привязка к системе/участку` -> `work_log.scope_ref` (`HYPOTHESIS`)

## 3) Реестр сертификатов/файлы сертификатов -> Certificate
1. `Наименование материала/оборудования` -> `certificate.material_links[]` (`HYPOTHESIS`)
2. `Номер/идентификатор сертификата` -> `certificate.certificate_id` (`HYPOTHESIS`)
3. `Файл сертификата` -> `certificate.certificate_file` (`CONFIRMED` как сущность файла)

## 4) Бизнес-правила связки (`CONFIRMED`/`OBSERVED`)
1. Даты актов сверяются с журналом работ.
2. Допустимы отклонения дат 1-2 дня, если нет явных нестыковок.
3. Дата окончания работ не должна быть позже закрытия.
4. Материалы должны иметь связанный сертификат по реестру.
5. Если сертификатов на акт 6 и более, формируется приложение.
6. Если сертификатов 5 и меньше, реквизиты указываются в акте.
7. Выход формируется как отдельный Word-файл на каждый акт.
8. Именование файла: `№<номер>-<система>_<секция>.docx` плюс приписка типа при необходимости.
9. Исходные акты в `ready-acts` полностью read-only: не править содержимое и не переименовывать файлы в `inputs/`.

## 5) Что нужно подтвердить входами
1. Точные названия колонок в реестрах.
2. Правила обработки пропущенных/дублирующихся строк.
3. Формальные правила связки «материал -> сертификат», если названия не совпадают дословно.

## 6) Рабочая таблица реестров (зафиксировано, 2026-03-28)
1. Рабочая таблица ведётся отдельно от журнала работ (`CONFIRMED`).
2. Рабочая таблица хранится как производный артефакт вне `inputs/`, в `output/spreadsheet/` (`CONFIRMED`).
3. Обязательные листы рабочей таблицы (`CONFIRMED`):
- `База актов`
- `Реестр (акты)`
- `Реестр (схемы)`
- `Реестр (сертификаты)`
4. Лист `База актов` является главным рабочим листом для дальнейших срезов по типам актов и секциям (`CONFIRMED`).
5. Порядок колонок листа `База актов` слева направо (`CONFIRMED`):
- `Шаблон`
- `Секция`
- `№ акта`
- `Тип акта`
- `Наименование акта`
- `Последующая работа`
- `Дата начала работ`
- `Дата завершения работ`
- `Дата оформления`
- `Примечания`
6. Порядок колонок листа `Реестр (акты)` слева направо (`CONFIRMED`):
- `Шаблон`
- `Секция`
- `№ акта`
- `Тип акта`
- `Наименование акта`
- `Дата оформления`
- `Примечания`

- `Наименование акта по реестру`
7. Допустимые значения поля `Шаблон` для текущего ВК (`CONFIRMED`):
- `АОСР`
- `АОУСИТО`
- `гидравлика`
- `дезинфекция`
- `испытания оборудования`
- `испытание на водоотдачу`
- `испытание АПТ`
- `пролив`
- `промывка`
8. Цветовая маркировка листа `Реестр (акты)` (`CONFIRMED`):
- `АОСР`: без заливки
- `АОУСИТО`: без заливки
- `дезинфекция`: светло-оранжевый
- `промывка`: светло-голубой
- `испытания оборудования`: светло-розовый
- `гидравлика`: светло-фиолетовый
- `пролив`: светло-жёлтый
- Готовые акты из `inputs/.../ready-acts/`: светло-зелёный с приоритетом над категорийной заливкой.
- Добавленные дубликаты номеров (например, `№6.1-...`): ярко-синий.

## 7) Выходные копии ready-acts (зафиксировано, 2026-03-27)
1. Все проверки и переименование готовых актов выполняются только на копиях в `output/vk/projects/<project>/...` (`CONFIRMED`).
2. В `output` формируется только рабочая структура секций с актами; `shared`-контент из `inputs` не дублируется (`CONFIRMED`).
3. Копии актов раскладываются по папкам систем (типов актов) согласно реестру; для вариативных строк внутри одной группы используется папка группы по заголовку реестра (`CONFIRMED`).
4. Если акту не найдена пара в реестре (лишний акт), он размещается в корне секции и в имени получает префикс `! `, затем номер из содержимого (с fallback на имя файла) и, при распознавании, тег типа акта (`CONFIRMED`).
5. Если по одному номеру найдено несколько исходных файлов, основной номер остаётся у акта, который совпадает с реестром по заголовку; дополнительный акт включается в комплект как дубликат с суффиксом номера (`№6.1-...`) и отдельной строкой в `Реестр (акты)` (`CONFIRMED`).
