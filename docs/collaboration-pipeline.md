# Collaboration Pipeline

Как работаем по проекту по шагам, чтобы не терять контроль над логикой и качеством.

## Короткий ответ
- Да, после каждого шага ты смотришь промежуточный результат в файлах.
- Мы двигаемся итерациями: шаг -> фиксация в docs -> твоя проверка -> следующий шаг.

## Рекомендуемый режим работы
1. Берём ограниченный срез данных (обычно 1 секция проекта).
2. Я вношу изменения только под цель текущего шага.
3. Фиксирую результат в документации и явно отмечаю:
- что подтверждено;
- что остаётся гипотезой;
- какие есть открытые вопросы.
4. Ты смотришь промежуточный результат и даёшь правки/подтверждение.
5. После подтверждения переходим к следующему шагу.

## Порядок этапов
1. Загрузка минимального боевого пакета (реестр, журнал, сертификаты, шаблоны) по одной секции.
2. Калибровка структуры полей и правил на этом пакете.
3. Закрытие спорных моментов и обновление rules/docs.
4. Проверка на 2-3 секциях (масштабирование правил).
5. Переход к реализации автоматизации (Python) только после стабилизации правил.

## Что ты проверяешь после каждого шага
1. Соответствует ли результат реестру и твоей практике.
2. Нет ли лишних допущений.
3. Корректно ли отражены исключения и оговорки.
4. Полно ли зафиксированы вопросы в `docs/open-questions.md`.

## Что я обязан отдавать в конце каждого шага
1. Список изменённых файлов.
2. Кратко: что изменено и зачем.
3. Что нужно проверить тебе.
4. Какие вопросы блокируют следующий шаг.

## Где фиксируем изменения
- Общая логика: `docs/process-overview.md`, `docs/algorithm-draft.md`.
- Наблюдения и рутина: `docs/process-observations.md`.
- Источники приоритета: `docs/sources-of-truth.md`.
- Открытые вопросы: `docs/open-questions.md`.
- Правила ВК: `sections/VK/section-rules.md`, `sections/VK/field-mapping.md`, `sections/VK/act-types.md`.

## Принцип качества
- Не пытаемся сделать «сразу всё».
- Сначала точность и воспроизводимость на реальных данных.
- Потом масштабирование и автоматизация.
