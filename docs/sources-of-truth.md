# Sources Of Truth

Иерархия доверия к источникам в этом репозитории.

## Уровни доверия
1. Прямые новые указания пользователя (`HIGHEST`)
- Самый приоритетный источник, перекрывает старые решения и исторические файлы.

2. Реальные актуальные входные и выходные примеры текущего проекта
- Фактические документы, на которых выполняется работа сейчас.
- Для ВК приоритет внутри входов: примечания в реестре важнее, журнал используется для сверки дат.
- Если в `output/.../work-log/` есть вручную сверенный производный файл журнала текущего проекта, для расшифровки текста и дат он приоритетнее сырой копии журнала из `inputs/`, так как `inputs/` остаётся read-only и исправления фиксируются только в производном артефакте.
- Для реквизитов шаблонов (название объекта, ФИО подписантов, номера приказов, ИНН/ОГРН и др.) эталон: существующие акты `Промывка` и `Дезинфекция`.

3. Утверждённые rules-файлы раздела (`sections/<SECTION>/*.md`)
- Правила, зафиксированные после согласования.

4. Референсные похожие проекты (`references/example-projects/`)
- Источник паттернов структуры и именования, но не безусловная норма.

5. Черновой алгоритм и обзор процесса в `docs/`
- Рабочая модель, которую можно пересматривать.

6. Исторические заметки (`references/obsidian/raw/`) (`LOWEST`)
- Полезный контекст и наблюдения, но не финальная спецификация.

## Почему такая иерархия
- Цель проекта: адаптация под живой процесс, а не фиксация старой практики.
- Исторические заметки могут содержать устаревшие/противоречивые решения.
- Референс-проекты показывают «как было», но не всегда «как должно быть сейчас».

## Правило разрешения конфликтов
1. Зафиксировать конфликт в `docs/open-questions.md`.
2. Сослаться на обе конфликтующие версии.
3. Не применять ни одну версию молча.
4. Дождаться подтверждения от пользователя и обновить docs/sections.

## Специализированный приоритет для ВК (текущий проект)
1. Состав актов, номера и типы актов: строго по реестру.
2. Даты: в первую очередь по примечаниям реестра, журнал использовать для сверки (допуск 1-2 дня без явных нестыковок).
3. Для `АОСР` дата начала работ по системе не должна быть раньше даты начала работ соответствующего `АОУСИТО`.
4. Дату начала работ `АОУСИТО` перед использованием сверять с существующим актом `АОУСИТО`; значение в примечании рабочей таблицы должно совпадать с датой в акте.
5. Объект, подписанты, приказы, реквизиты организаций: по существующим актам `Промывка` и `Дезинфекция`.
6. АОУСИТО не использовать для сверки объекта, подписантов, приказов и реквизитов.
7. Для колонки `Пределы по АОУСИТО` диапазон начала/окончания работ брать из существующего акта `АОУСИТО` по соответствующей системе, а не из предположений.
8. Записи журнала о бурении отверстий, сверлении и сходной подготовке не использовать как подтверждение монтажа трубопроводов или других элементов системы.
