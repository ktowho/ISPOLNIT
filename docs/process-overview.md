# Process Overview (Draft)

Документ описывает текущий рабочий контур процесса для ВК и специально не фиксирует его как окончательный регламент.

## Легенда статусов
- `CONFIRMED`: подтверждено прямыми указаниями пользователя.
- `OBSERVED`: устойчиво наблюдается в исторических заметках или референс-примерах.
- `HYPOTHESIS`: рабочее предположение до получения реальных входов текущего проекта.

## Цель процесса
Подготовить и собрать комплект исполнительной документации по ВК с возможностью будущей автоматизации.

## Контур процесса
1. Приём входов (`CONFIRMED`)
- Принимаются Word-реестры, журнал работ, материалы/оборудование, сертификаты, шаблоны актов.
- Исходные акты принимаются как неизменяемый вход: в `inputs/` запрещено менять и содержимое, и имена файлов.
- Любые проверки, переименование и нормализация выполняются только на копиях в `output/`.

2. Извлечение данных из реестров (`CONFIRMED`)
- Данные из Word-реестров должны извлекаться в структурированном виде.

3. Нормализация данных в единую внутреннюю структуру (`CONFIRMED`)
- Сущности должны быть согласованы между собой: акты, системы, материалы, сертификаты, листы ИС, даты.

4. Обогащение и сверка (`CONFIRMED` + `OBSERVED`)
- Подстановка дат по журналу работ (`CONFIRMED`).
- Привязка материалов к сертификатам (`CONFIRMED`).
- Частые проверки комплектности и правки после замечаний (`OBSERVED`).
- Приоритет при конфликте: примечания в реестре важнее (`CONFIRMED`).
- Отклонения дат на 1-2 дня допустимы при отсутствии явных нестыковок (`CONFIRMED`).
- Дата окончания работ не должна быть позже закрытия (`CONFIRMED`).

5. Формирование актов (`CONFIRMED` + `OBSERVED`)
- Генерация актов по Word-шаблонам (`CONFIRMED`).
- Разделение: 1 акт = 1 Word-файл (`CONFIRMED`).
- Если на акт 6 и более сертификатов, формируется приложение (`CONFIRMED`).
- Если сертификатов 5 и меньше, реквизиты указываются в самом акте (`CONFIRMED`).
- Перед выпуском шаблоны сверяются с существующими актами `Промывка` и `Дезинфекция` по объекту, подписантам, приказам и реквизитам организаций (`CONFIRMED`).

6. Правила состава и именования (`CONFIRMED`)
- Состав актов брать строго из реестра, без отклонений (кроме явных грамматических опечаток).
- Базовый шаблон имени файла: `№<номер>-<система>_<секция>.docx`.
- Для приложений/испытаний/промывки и т.п. добавлять приписку в конце имени.
- Для дат использовать формат по месту документа, ориентируясь на примеры конкретного поля.

7. Сборка и рабочая таблица (`CONFIRMED` + `OBSERVED`)
- Работать можно в одной таблице, но с разбиением по секциям сверху вниз по реестрам (`CONFIRMED`).
- В референсах встречается упаковка актов, сертификатов, схем в итоговый комплект по секции (`OBSERVED`).
- В текущем этапе схемы пользователь готовит вручную (`CONFIRMED`).

8. Масштабирование на другие разделы (`CONFIRMED`)
- В будущем контур должен расширяться на OV1 и OV2 без зашивки различий в общее ядро.

## Что уже стабильно видно по рабочим данным
- Повторяется цикл: реестры -> таблица -> даты по журналу -> материалы/сертификаты -> печать/разделение -> доработки.
- Наиболее трудоёмкие места: заполнение материалов/сертификатов, ручная компоновка схем, исправления по замечаниям.

## Что пока нельзя считать нормой
- Конкретные названия колонок реестра.
- Жёсткий порядок шагов без вариантов.
- Окончательные правила именования актов и комплектов.
