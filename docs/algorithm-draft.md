# Algorithm Draft (Черновик)

Документ фиксирует рабочий алгоритм в изменяемом виде. Любой шаг может быть изменён, переставлен, объединён или удалён.

## Легенда
- `CONFIRMED`: прямое требование пользователя.
- `OBSERVED`: повторяется в заметках/примерах.
- `HYPOTHESIS`: требует подтверждения на реальных текущих входах.

## Черновой скелет
1. `A01` Сбор входов по проекту (`CONFIRMED`)
- Входы: реестры Word, журнал работ, материалы/оборудование, сертификаты, шаблоны актов, пример итогового комплекта.
- Выход: проверенный набор исходных файлов.

2. `A02` Первичная инвентаризация файлов (`OBSERVED`)
- Проверить наличие дублей, черновых временных файлов, неполных документов.
- Исходные акты считать полностью read-only: запрещено менять содержимое и имя файла в `inputs/`; любые правки делать только на копиях в `output/` (`CONFIRMED`).

3. `A03` Извлечение данных из Word-реестров (`CONFIRMED`)
- Перенос содержимого в промежуточную структуру для дальнейшей нормализации.

4. `A04` Нормализация данных (`CONFIRMED`)
- Выравнивание идентификаторов актов/систем/материалов.
- Разделение дат (например, дата завершения работ и дата подписания) при необходимости (`OBSERVED`).

5. `A05` Сверка с журналом работ (`CONFIRMED` + `OBSERVED`)
- Подстановка/проверка дат выполнения работ.
- Фиксация расхождений между реестром и журналом.
- Приоритет при конфликте: примечания реестра.
- Допустимы отклонения 1-2 дня при отсутствии явных нестыковок.
- Дата окончания работ не позже даты закрытия.

6. `A06` Привязка материалов и сертификатов (`CONFIRMED`)
- Для каждого акта определить перечень материалов и ссылки на подтверждающие сертификаты.
- Использовать сертификаты, указанные в реестре; дополнительные файлы в папке не обязательны для обработки.

7. `A07` Проверка порога приложений (`CONFIRMED`)
- Если на один акт 6+ сертификатов, формировать приложение.
- Если сертификатов 5 и меньше, реквизиты перечислять в самом акте.

8. `A08` Подготовка выходных таблиц (`CONFIRMED`)
- Разложить нормализованные данные по Excel-таблицам рабочего контура.

9. `A09` Формирование актов по Word-шаблонам (`CONFIRMED`)
- Подставить данные в шаблоны.
- Обработать специальные типы актов при наличии (`OBSERVED`).
- Типы и состав актов брать строго из реестра (кроме явной грамматической опечатки).
- Именование: `№<номер>-<система>_<секция>.docx`, с припиской типа для приложений/испытаний/промывки и т.п.
- Перед выпуском сверить шаблоны с существующими актами `Промывка` и `Дезинфекция` по полям: объект, ФИО подписантов, номера приказов, реквизиты организаций (`CONFIRMED`).

10. `A10` Разделение на отдельные акты (`CONFIRMED`)
- Правило: один акт = один Word-файл.

11. `A11` Контроль комплектности (`OBSERVED`)
- Проверить полноту: акты, приложения, сертификаты, схемы, даты, связи.

12. `A12` Сборка итогового комплекта (`OBSERVED`)
- Подготовить структуру финальной выдачи по согласованному формату.
- Рабочий табличный контур может быть единым, но с разбиением по секциям сверху вниз по реестрам.

## Гибкость алгоритма
- Шаги `A02`, `A08`, `A11`, `A12` допускают перестановку.
- Шаг `A07` зависит от подтверждённого порога 6+ сертификатов.
- Любые новые исключения и ветки процесса вносятся как отдельные пункты, а не «поправки в уме».

## Ограничения
- Пока нет права придумывать предметные правила при нехватке входных данных.
- Любое сомнение фиксируется в `docs/open-questions.md`.
