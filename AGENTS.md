# AGENTS.md

## Назначение репозитория
Этот репозиторий — база для поэтапной подготовки автоматизации оформления исполнительной документации по строительному разделу ВК.

## Текущий этап
- Сейчас этап подготовки базы: структура, правила, нормализованная документация, сбор и декомпозиция знаний.
- На этом этапе не писать production-код и не реализовывать пайплайн.

## Статус алгоритма
- Текущий алгоритм процесса — рабочий черновик.
- Алгоритм будет пересматриваться: шаги можно менять, переставлять, добавлять и удалять.
- Нельзя считать текущую схему окончательным регламентом.

## Работа с источниками
- Исторические заметки (например, `references/obsidian/raw/`) — вспомогательный источник, а не финальная спецификация.
- Референсные примеры прошлых проектов (например, `references/example-projects/`) — источник наблюдений и паттернов, но не абсолютный стандарт.
- При нехватке данных нельзя придумывать предметную логику.
- Если источники противоречат друг другу, расхождения фиксируются явно, без молчаливого выбора версии.
- Для текущего проекта эталон по названию объекта, ФИО подписантов, номерам приказов и реквизитам организаций (ИНН/ОГРН и т.п.) берётся из существующих актов `Промывка` и `Дезинфекция`.
- АОУСИТО не использовать как источник сверки для этих данных.

## Работа с исходными актами
- Исходные акты в `inputs/` полностью read-only: запрещено менять содержимое и имена файлов.
- Любые проверки, переименование и нормализация выполняются только на копиях в `output/`.
- Исходные файлы в `inputs/` используются только как источник данных для сверки и копирования.

## Обязательная фиксация неопределённостей
- Все спорные места, противоречия и недостающие входы фиксировать в `docs/open-questions.md`.

## Архитектурные границы
- Различия по разделам (ВК, OV1, OV2 и т.д.) нельзя зашивать в общее ядро.
- Общие правила держать отдельно от правил конкретного раздела.

## Порядок работы перед изменениями
Перед любыми изменениями обязательно читать:
- документы в `docs/`;
- файлы в `sections/` (включая подразделы, например `sections/VK/`).

## Правило синхронизации документации
- Любое изменение логики процесса должно сопровождаться обновлением документации.
- Нельзя оставлять старые схемы как будто они неизменны.

## Классификация утверждений в документации
В каждом релевантном документе отделять:
- подтверждённые правила;
- рабочие гипотезы;
- наблюдения из примеров;
- черновые шаги;
- открытые вопросы.

## Запреты текущего этапа
- Не создавать фейковые примеры данных.
- Не писать код (Python/Node.js и т.п.) для реализации процесса.
- Не строить избыточно сложную архитектуру заранее.
